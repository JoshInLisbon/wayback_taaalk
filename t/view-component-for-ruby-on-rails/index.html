ViewComponents are different from traditional Rails views.Â 
A ViewComponent is a Ruby file and corresponding template file with the same base name:
app/components/test_component.rb:
class TestComponent < ViewComponent::Base
  def initialize(title:)
    @title = title
  end
end
app/components/test_component.html.erb:
<span title="<%= @title %>"><%= content %></span>
Rendered in a view as:
<%= render(TestComponent.new(title: "my title")) do %>
  Hello, World!
<% end %>
Returning:
<span title="my title">Hello, World!</span>
Most importantly, ViewComponents are encapsulated: the template code executes within the context of the Ruby object, unlike traditional Rails views which execute in a global context.

Our component unit tests follow a consistent pattern: render the component, then assert against the DOM output using Capybara matchers:
render_inline(MyComponent.new)

assert_selector("h1", text: "Hello, World!")
For comparison, we previously tested our views exclusively through controller tests, which rendered the whole action, then asserted against the rendered DOM:
get "/user/repository"

assert_selector("h2", text: "My Cool Repo")
While the tests look very similar, the component unit test is over 100x faster than the controller unit test (50ms vs. 5 seconds). Also, since the component is tested directly, we don't have to test it for every place it's used, unlike a shared partial covered by controller tests.Â
